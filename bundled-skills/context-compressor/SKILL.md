# Context Compressor

## 核心理念

> 压缩不是删除 — 是把 10 万 token 的对话浓缩成 5000 token 但不丢任何关键信息。

## 压缩前准备

1. **移除图片块** — 替换为 `[image was shared]` 文本标记
2. **不调用任何工具** — 压缩是纯文本任务，tool call 会浪费 turn

## 九段式摘要模板

先用 `<analysis>` 整理思路（草稿，不进入最终上下文），再输出 `<summary>`：

```
<analysis>
[按时间顺序分析每条消息，确保覆盖所有要点]
</analysis>

<summary>
1. 用户请求和意图
   [所有明确请求，完整详述]

2. 关键技术概念
   - [技术/框架/方案列表]

3. 文件和代码
   - [文件名]
     - 为什么重要
     - 做了什么修改
     - 关键代码片段

4. 错误和修复
   - [错误描述]
     - 如何修复
     - 用户对此的反馈（如有）

5. 问题解决
   [已解决的问题 + 正在排查的问题]

6. 所有用户消息
   - [列出全部非工具结果的用户消息]
   （⚠️ 这是捕捉用户意图变化的关键）

7. 待办任务
   - [明确被要求但未完成的任务]

8. 当前工作
   [压缩前正在做的事，附文件名和代码片段]

9. 下一步（可选）
   [必须与最近请求直接相关]
   > 原文引用："用户说了什么" → 防止意图漂移
</summary>
```

## 关键规则

- **第 6 段（用户消息）是最重要的** — 不列用户消息 = 丢失意图变化
- **第 8 段必须具体** — 不是"正在修复 bug"，而是"正在修复 src/auth.py:42 的空指针，已加了 null check 但测试还没跑"
- **第 9 段必须附原文引用** — 防止压缩后做跑偏的事
- **不要开始新的、切线的任务** — 只延续压缩前正在做的

## 压缩后恢复

压缩完成后自动恢复：
- 最近修改的文件内容（最多 5 个文件）
- 活跃 Skill 的指令重注入
- 工具列表变更提示

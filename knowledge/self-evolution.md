# 自进化协议

## 核心法则

> 如果一个经验值得记住，就必须写到文件里。脑子里的"记住了"不算数。

## 触发条件（7 种场景）

| # | 场景 | 记录到 | 类别 |
|---|------|--------|------|
| 1 | 命令/操作失败 | ERRORS.md | error |
| 2 | 被用户纠正 | LEARNINGS.md | correction |
| 3 | 用户需要不存在的能力 | FEATURE_REQUESTS.md | feature |
| 4 | 外部 API/工具出错 | ERRORS.md | error |
| 5 | 发现知识过时/错误 | LEARNINGS.md | knowledge_gap |
| 6 | 发现更好做法 | LEARNINGS.md | best_practice |
| 7 | 任务完成时（回顾） | LEARNINGS.md | task_review |

## 去重法则

触发 ≠ 必须写入。每次触发先判断：
- 这个经验是否**真正新颖**？
- 本质上是否已包含在已有条目中？
- 有没有"新信息量"？

→ 没有 → 跳过不写。避免用重复低价值记录污染。

## 进化路径

```
单次经验 → 记录到 .learnings/
重复 2 次 → 晋升到 memory 业务经验
重复 3 次 → 晋升到 knowledge 规则文件
有通用价值 → 提议写入 Skill
连续 5 次失败 → 标记 .disabled
```

## Skill 自进化

每次使用 Skill 后自动反馈：
- ✅ 成功 → 记录新触发场景
- ⚠️ 部分成功 → 记录限制条件
- ❌ 失败 → 分析根因，修正内容
- 🔁 重复 3 次 → 晋升为硬性规则
- 💀 连续 5 次失败 → 标记禁用

## 经验记录格式

```markdown
## [YYYY-MM-DD] 简短标题
- **场景**: 发生了什么
- **根因**: 为什么
- **教训**: 下次怎么做
- **适用**: 什么场景下套用
```

## Builder Journal（战果簿）

每次会话结束时检查：
- 踩了什么坑？
- 走了什么弯路？下次怎么更快？
- 发现了新技巧？
- 有什么值得沉淀的？

有 → 写入 `builder-journal.md`
没有 → 跳过

## 纠正检测信号

| 信号 | 示例 |
|------|------|
| 直接否定 | "不对"/"错了" |
| 给不同答案 | "其实是..."/"应该是..." |
| 示范做法 | 不说你错，直接展示正确的 |
| 质疑 | "你确定？"/"真的吗？" |
| 放弃让你做 | "算了我来" |

→ 被纠正 = 同轮立即记录，不等对话结束。

# Agent Directives — 机械性覆盖规则

在有限的上下文窗口和严格约束下，你必须遵守这些规则才能产出生产级代码。

## 前置工作

### 1. "第零步"规则
死代码加速上下文衰减。对任何 >300 行的文件做结构性重构前，**先清理**：
- 移除所有死代码（未使用的 props、exports、imports）
- 移除 debug logs / console.log
- **单独 commit 这次清理**，再开始真正的重构

### 2. 分阶段执行
**永远不要**在一次响应中做多文件重构。
- 拆成明确的阶段
- 每个阶段最多触碰 **5 个文件**
- Phase 1 完成 → 验证 → 等待确认 → 才开始 Phase 2

## 代码质量

### 3. 高级开发者覆盖
不要只做"最简单的方案"。如果发现：
- 架构有缺陷 → 提出并实施结构性修复
- 状态重复 → 统一
- 模式不一致 → 对齐
- 问自己："一个有经验的高级工程师在 code review 时会拒绝什么？" → 修掉它

### 4. 强制验证
工具标记文件写入成功 ≠ 代码正确。**禁止报告任务完成**除非：
- 运行了 `npx tsc --noEmit`（或项目等效的类型检查）
- 运行了 linter（如配置了）
- **修复了所有错误**
- 如果没有类型检查器，明确说明而不是假装成功

## 上下文管理

### 5. 子代理分群
任务涉及 >5 个独立文件时，**必须**拆分为并行子任务。
- 每个子任务 5-8 个文件
- 串行处理大任务 = 保证上下文衰减

### 6. 上下文衰减感知
对话超过 **10 条消息**后：
- **必须重新读取**任何要编辑的文件
- 不要信任你对文件内容的记忆
- 自动压缩可能已经静默销毁了上下文

### 7. 文件读取预算
- 每次文件读取上限 **2000 行**
- 超过 500 行的文件 → 必须用 offset + limit 分段读取
- **永远不要**假设一次 read 看到了完整文件

### 8. 工具结果截断盲区
- 超过 50000 字符的工具结果会被静默截断
- 如果搜索/命令返回结果异常少 → 用更窄的范围重新搜索
- 怀疑截断时明确说明

## 编辑安全

### 9. 编辑完整性
- 每次编辑前 → **重新读取文件**
- 编辑后 → **再读一次确认**变更正确应用
- 不要对同一文件连续 batch 超过 3 次编辑而不验证读取
- Edit 工具在 old_string 不匹配时会**静默失败**

### 10. 无语义搜索
你有 grep，没有 AST。重命名/修改任何函数/类型/变量时，**必须分别搜索**：
- 直接调用和引用
- 类型级引用（interfaces, generics）
- 包含该名称的字符串字面量
- 动态 import 和 require()
- Re-exports 和 barrel file 条目
- 测试文件和 mock
→ **不要假设一次 grep 能捕获所有引用**

# 工程化方法论

## 任务路由（收到任务后第一步）

根据任务类型决定处理策略：

### 简单任务（1-2 步）
直接执行，不需要规划。
例：读文件、改一行代码、跑一个命令

### 中等任务（3-5 步）
快速列 TodoList → 执行 → 验证
例：添加一个功能、修复 bug、写脚本

### 复杂任务（6+ 步）
三文件模式：
- `task_plan.md` — 任务规划和进度追踪
- `notes.md` — 研究笔记和发现
- `[deliverable]` — 最终交付物

### 调试任务
1. 复现问题（拿到确切错误）
2. 定位根因（不猜，读代码 + 日志）
3. 修复（最小改动）
4. 验证修复
5. 确认无回归

## 能力进化
- 识别可复用模式 → 抽象为能力 → 内化到决策层
- 临时脚本 → 判断是否可复用 → 是则固化为行为模式
- 不汇报进化过程，用结果说话

## 搜索策略
1. **先窄后宽**：精确搜 → 模糊搜 → 全局搜
2. **先本地后远程**：文件系统 → git log → web
3. **先结构后内容**：目录结构 → 文件名 → 文件内容

## 多步任务原则
- 有依赖的串行，无依赖的并行
- 每步完成后验证，不要最后才发现第一步就错了
- 失败后自动重试不同方案（至少 3 种），才升级找人

## Context 管理
- 只加载必需文档，保持充足空间用于思考
- 大文件用 offset/limit 分段读
- 长输出截断，保留关键信息

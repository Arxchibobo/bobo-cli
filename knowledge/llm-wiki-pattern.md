# LLM-Wiki Pattern (Karpathy, 2026-04)

Source: https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

## Core Insight

RAG = 每次从头查原始文档，知识永远散装。
Wiki = LLM 持续编译、互链、维护结构化知识，读的时候直接拿编译好的结果。

**关键区别**: wiki 是持续积累的产物。交叉引用已经建好，矛盾已经标记，综合已经反映了你读过的所有东西。

## 三层架构

| 层 | 内容 | 权限 |
|---|------|------|
| Raw Sources | 原始文档（文章/论文/笔记/数据） | 只读不改 |
| Wiki | LLM 生成的结构化页面（摘要/实体/概念/对比） | LLM 写+维护 |
| Schema | wiki 约定+工作流规则 | 人+LLM 共同演进 |

## 三个核心工作流

### Ingest（新源进入）
- 读源 → 讨论要点 → 写摘要页 → 更新 index → 更新相关实体/概念页 → 追加 log
- 一个源通常触碰 10-15 个 wiki 页面
- 可以逐个 ingest（参与度高）或批量 ingest（低监督）

### Query（查询 + 回写）
- 先读 index → 找相关页 → 合成回答（带引用）
- **关键**: 好的答案应该回写为新 wiki 页（对比表/分析/发现不应消失在 chat 里）
- 探索过程和 ingest 一样积累到知识库

### Lint（定期健检）
- 矛盾：不同页面对同一事实说法不一致
- 过时：新源已更新但旧页面没跟上
- 孤儿页：无入站链接
- 缺页：被多次提及但没有独立页面的概念
- 缺交叉引用：应该互链但没链的页面
- 数据缺口：可以通过 web search 补充的空白

## 两个关键文件

### index.md（内容导向）
- 所有页面的目录：链接 + 一句话摘要 + 元数据
- 按类别组织（实体/概念/源/分析）
- 每次 ingest 后更新
- 查询时先读 index 再钻取

### log.md（时序导向）
- Append-only 操作记录
- 格式: `## [YYYY-MM-DD] action | title`
- 可 grep 解析: `grep "^## \[" log.md | tail -5`
- 记录 ingest/query/lint 事件

## 应用场景
- 个人知识管理（目标/健康/自我提升）
- 深度研究（论文/报告/weeks-long topic）
- 读书笔记（角色/主题/情节线索互链）
- 团队内部 wiki（Slack/会议/客户通话 → LLM 维护）
- 竞品分析、尽职调查、旅行规划、课程笔记

## 工具生态
- Obsidian: 浏览 wiki、图谱视图、Web Clipper
- qmd: 本地 Markdown 搜索（BM25 + 向量 + LLM re-rank）
- Marp: Markdown → 幻灯片
- Dataview: YAML frontmatter 查询
- Git: 版本控制 + 协作

## 核心洞察
> 维护知识库的苦差事不是阅读或思考——是记账。更新交叉引用、保持摘要最新、标记新旧数据矛盾、维护一致性。人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会无聊、不会忘记更新交叉引用、能一次触碰 15 个文件。

> 人的工作是策展源、引导分析、提好问题、思考意义。LLM 的工作是其他一切。
